<h1 align="center">Микшер громкости</h1>

<p align="center">
  Громкость каждого приложения по отдельности — как в Windows, только на macOS.<br>
  <em>Per-app volume mixer for macOS. No drivers — built on the official CoreAudio process-tap API.</em>
</p>

---

**English summary:** the volume mixer macOS never shipped. A menu bar panel with a
volume slider, mute button and live VU meter for every app that plays audio, plus a
master volume slider. Per-app levels persist across restarts. Built on the official
CoreAudio process taps API (macOS 14.4+) — no kernel extensions, no virtual audio
drivers. Swift + SwiftUI, builds with Command Line Tools only (`./build.sh`).
Requires macOS 15+.

---

## Что умеет

- 🎚 Ползунок громкости 0–100% и mute для каждого приложения
- 📊 VU-метр — видно, кто именно шумит
- 🔇 Секция «Молчат»: настрой громкость Telegram *до* того, как он пикнет
- 🔊 Мастер-ползунок общей громкости системы
- 💾 Настройки запоминаются по bundle ID и переживают перезапуски
- 🪶 Никаких драйверов и расширений ядра — официальный API CoreAudio

## Установка

**Готовая сборка:** скачай DMG из [Releases](../../releases), перетащи приложение
в «Программы». macOS при первом запуске скажет «разработчик не подтверждён» —
это нормально для сборки без платного аккаунта Apple: разреши запуск в
Системные настройки → Конфиденциальность и безопасность → «Всё равно открыть».

**Из исходников** (нужны только Command Line Tools, Xcode не обязателен):

```bash
./build.sh                          # соберёт build/Микшер громкости.app
open "build/Микшер громкости.app"
```

При первом запуске macOS попросит разрешение на **запись системного звука**.
Без него регулировать громкость приложений технически невозможно: перехват
звука в macOS проходит через этот механизм. Микшер ничего не записывает и
никуда не отправляет — весь код открыт, можешь убедиться.

Требуется macOS 15 или новее (API process taps появился в 14.4).

## Как это работает

На каждое играющее приложение создаётся muted process tap
(`AudioHardwareCreateProcessTap`) — система глушит оригинальный вывод процесса
и отдаёт его аудиопоток нам. Поток проходит через приватное агрегатное
устройство: сэмплы умножаются на коэффициент громкости (перцептивная
квадратичная кривая) и уходят в текущее устройство вывода. Там же считается
RMS для VU-метра. Смена устройства вывода пересоздаёт цепочку автоматически.

Забавный факт, добытый отладкой: macOS не доставляет события изменения
`kAudioProcessPropertyIsRunningOutput`, поэтому список играющих приложений
обновляется лёгким поллингом раз в 1.5 с — иначе приложения, начавшие играть
после запуска микшера, оставались бы невидимыми.

## Разработка

```bash
./test.sh                           # тесты (swift-testing; обёртка нужна из-за CLT)
./build.sh                          # релизная сборка .app
./dist.sh                           # DMG для раздачи
```

Структура:

- `Sources/VolumeMixerCore` — логика: монитор аудиопроцессов, tap-контроллеры,
  движок, настройки, системная громкость
- `Sources/VolumeMixerApp` — SwiftUI: MenuBarExtra-панель, строки приложений,
  онбординг разрешения
- `docs/superpowers/` — дизайн-документ и план, по которым это строилось

## Ограничения v1

- Все приложения играют в текущее устройство вывода — роутинга по устройствам нет
- Эквалайзера и горячих клавиш нет
- Сборка подписывается ad-hoc — для распространения без предупреждений macOS
  нужен Apple Developer ID и нотаризация
